# StoryTree 权限系统重构实施总结

## 📅 实施日期
2026-03-12

## 🎯 重构目标
根据用户需求，重构权限系统以实现：
1. 使用单向关注替代双向好友系统
2. 简化协作者角色（移除owner/editor/viewer三种角色）
3. 明确分支创作权限（仅协作者可续写）
4. 统一权限粒度

### 2. 权限检查逻辑重写 (api/src/utils/permissions.ts)

#### 2.1 新增基础权限函数
```typescript
isStoryAuthor(userId, storyId)       // 检查是否是主创作者
isCollaborator(userId, storyId)      // 检查是否是协作者（未被移除）
isFollowingAuthor(userId, storyId)   // 检查是否关注了故事作者
nodeHasChildren(nodeId)              // 检查章节是否有子节点
```

#### 2.2 更新权限检查函数

**canViewStory** - 查看故事权限
- `public`: 任何人可见
- `friends`: 关注者、协作者、作者可见
- `private`: 仅作者和协作者可见

**canEditStory** - 编辑故事权限
- 仅主创作者可以编辑故事设置

**canCreateBranch** - 创建分支权限
- 主创作者始终可以
- 协作者根据`allow_branch`设置决定
- 其他人不可以

**canComment** - 评论权限
- 必须有查看权限
- 故事允许评论（`allow_comment = true`）

**canDeleteNode** - 删除章节权限
- 故事主创作者可以删除任何章节
- 章节作者可以删除自己的章节（有子节点需提交审核）

## 📊 权限矩阵对比

### 优化前 vs 优化后

| 场景 | 优化前 | 优化后 |
|------|--------|--------|
| 好友系统 | 双向确认（friendships表） | 单向关注（复用follows表） |
| 协作者角色 | owner/editor/viewer三种 | 仅协作者/非协作者 |
| 公开故事续写 | 任何登录用户可续写 | 仅协作者可续写 |
| 评论权限 | 独立检查allow_comment | 绑定可见性+allow_comment |
| 章节权限 | 继承故事权限（隐式） | 明确继承故事权限（显式） |
| 协作者移除 | 物理删除 | 软删除（保留历史） |

## 🎨 新的权限体系

### 用户角色定义
1. **主创作者（Author）**：故事的原创者
2. **协作者（Collaborator）**：被邀请参与创作的用户
3. **读者（Reader）**：普通访问者

### 权限详细说明

| 操作 | 主创作者 | 协作者 | 关注者 | 普通读者 |
|------|---------|--------|--------|---------|
| 查看私密故事 | ✅ | ✅ | ❌ | ❌ |
| 查看关注者可见故事 | ✅ | ✅ | ✅ | ❌ |
| 查看公开故事 | ✅ | ✅ | ✅ | ✅ |
| 编辑故事设置 | ✅ | ❌ | ❌ | ❌ |
| 创建分支续写 | ✅ | ✅* | ❌ | ❌ |
| 评论章节 | ✅** | ✅** | ✅** | ✅** |
| 删除章节 | ✅ | ✅*** | ❌ | ❌ |

*根据`allow_branch`设置  
**根据`allow_comment`设置且有查看权限  
***仅能删除自己的章节，有子节点需提交审核

## 🔧 技术实现亮点

### 1. 软删除机制
- 协作者被移除后不会物理删除记录
- 保留`removed_at`和`removed_by`字段
- 便于追溯历史和数据分析
- 支持重新激活协作者

### 2. 权限检查优化
- 独立的权限检查函数，易于测试和维护
- 清晰的权限层级：作者 > 协作者 > 关注者 > 读者
- 避免了角色判断的复杂性

### 3. 单向关注系统
- 复用已有的`follows`表
- 无需双向确认，降低用户操作成本
- 更符合现代社交产品的设计理念
