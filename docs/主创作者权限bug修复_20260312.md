# Bug修复：主创作者查看私密故事时分支图和统计分析不可见

## 🐛 问题描述

**发现时间：** 2026-03-12

**问题现象：**
当主创作者将故事可见性设置为"私密"或"关注者可见"时：
- ✅ 章节列表始终可见（正常）
- ❌ 分支图不可见（bug）
- ❌ 统计分析不可见（bug）
- ❌ 故事设置页面无法加载（bug）⭐ **新发现**

**问题影响：**
主创作者无法查看自己私密故事的分支图和统计分析，也无法打开故事设置页面，这不符合逻辑。作为故事的主创作者，应该对自己的故事拥有完全的查看权限，不受可见性设置的限制。

### 后端问题

#### 问题1: GET /:id/tree 路由

在`api/src/routes/stories.ts`的`GET /:id/tree`路由中：

**原代码（有问题）：**
```typescript
router.get('/:id/tree', optionalAuth, async (req, res) => {
  const { id } = req.params;
  const userId = getUserId(req);

  try {
    // ❌ 直接检查查看权限，没有区分主创作者
    const hasPermission = await canViewStory(userId, parseInt(id));
    if (!hasPermission) {
      return res.status(403).json({ error: '无权限查看此故事' });
    }

    const story = await prisma.stories.findUnique({
      where: { id: parseInt(id) },
      select: {
        id: true,
        title: true,
        root_node_id: true
        // ❌ 没有获取author_id，无法判断是否是主创作者
      }
    });
    // ...
  }
});
```

**问题分析：**
1. 权限检查在获取story信息之前执行
2. 没有获取`author_id`，无法判断当前用户是否是主创作者
3. `canViewStory`函数对私密故事返回false（即使是主创作者）
4. 导致主创作者也被403拒绝访问

## ✅ 解决方案

### 后端修复

#### 修复1: GET /:id/tree 路由

**修复后的代码：**

```typescript
router.get('/:id/tree', optionalAuth, async (req, res) => {
  const { id } = req.params;
  const userId = getUserId(req);

  try {
    // ✅ 先获取故事信息，包括author_id
    const story = await prisma.stories.findUnique({
      where: { id: parseInt(id) },
      select: {
        id: true,
        title: true,
        root_node_id: true,
        author_id: true  // ✅ 新增：获取作者ID
      }
    });

    if (!story) {
      return res.status(404).json({ error: 'Story not found' });
    }

    // ✅ 检查查看权限：主创作者始终有权限，其他用户需要检查
    const isAuthor = userId && story.author_id === userId;
    if (!isAuthor) {
      const hasPermission = await canViewStory(userId, parseInt(id));
      if (!hasPermission) {
        return res.status(403).json({ error: '无权限查看此故事' });
      }
    }
    // ...
  }
});
```

**修复要点：**
1. **调整权限检查顺序**：先获取故事信息，再进行权限检查
2. **获取author_id**：在select中添加`author_id`字段
3. **区分主创作者**：判断`userId === story.author_id`
4. **主创作者豁免**：如果是主创作者，跳过权限检查
5. **其他用户检查**：非主创作者才调用`canViewStory`进行权限检查

### 前端修复

**修复后的代码：**

```javascript
async function loadTreeData(storyId) {
    try {
        // ✅ 获取token（如果已登录）
        const token = localStorage.getItem('token') || sessionStorage.getItem('token');
        const headers = {};
        if (token) {
            headers['Authorization'] = `Bearer ${token}`;
        }

        // ✅ 请求时携带Authorization header
        const response = await fetch(`/api/stories/${storyId}/tree`, { headers });
        if (!response.ok) {
            const errorData = await response.json().catch(() => ({}));
            console.error('加载树状图失败:', response.status, errorData);
            throw new Error(errorData.error || '加载树状图数据失败');
        }

        const data = await response.json();
        window.treeData = data.tree;
        window.treeStats = data.stats;
        
        // 渲染统计卡片
        renderStatsCards(data.stats);
        
        console.log('树状图数据加载成功:', data);
    } catch (error) {
        console.error('加载树状图数据错误:', error);
    }
}
```

**修复要点：**
1. **获取token**：从localStorage或sessionStorage中获取登录token
2. **构建headers**：创建包含Authorization的请求头
3. **携带身份信息**：在fetch请求中传入headers
4. **改进错误处理**：更详细的错误日志输出

## 💡 经验总结

### 权限设计原则

1. **主创作者至上**：主创作者应该对自己的内容拥有完全控制权
2. **先身份后权限**：先判断用户身份（作者/协作者/读者），再检查权限
3. **合理豁免**：对于特殊角色（如主创作者），应该豁免某些权限检查
4. **一致性**：所有API接口的权限逻辑应该保持一致

### 类似问题排查

如果发现类似的权限问题，检查以下几点：

1. **权限检查时机**：是否在获取必要信息（如author_id）之前就进行了权限检查？
2. **角色判断**：是否正确判断了用户的角色（主创作者/协作者/读者）？
3. **权限豁免**：特殊角色是否有合理的权限豁免？
4. **API一致性**：所有相关API的权限逻辑是否一致？
